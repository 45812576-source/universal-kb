# Skill Studio 响应时延治理 — 顶层架构设计

日期：2026-04-15  
范围：`/Users/xia/project/universal-kb` + `/Users/xia/project/le-desk`  
状态：设计确认稿

## 1. 问题定义

如果以用户实际吐槽为准，则 Skill Studio 当前最突出的体验问题不是“最终答得对不对”，而是每轮对话等待时间过长，且等待过程缺乏确定性。

当前问题有三个层面：

- 用户不知道系统是否还在有效工作，还是只是卡住。
- 中高复杂请求常常要等到完整审计、治理、补充推理都结束，才拿到第一份可用反馈。
- 前后端当前链路里存在同步阻塞、重复装载和状态耦合，导致真实首答时延偏大。

本设计不接受“定时发两条安抚反馈”作为解决方案。  
如果工程链路本身没有变快，则单纯补中间提示不算达标。

## 2. 目标与硬约束

### 2.1 交互 SLA

如果请求被判定为中等复杂推理，则系统必须在 `30 秒` 内给出首份可用反馈。  

如果请求被判定为高等复杂推理，则系统必须在 `60 秒` 内给出首份可用反馈。

这里的“首份可用反馈”不是 loading，也不是“正在思考”，而必须至少包含：

- 当前判断已走到哪一步；
- 当前最主要的判断、结论或修复方向；
- 当前建议用户执行或等待的下一步。

### 2.2 范围约束

本次治理必须覆盖 Skill Studio 全场景，而不是只优化单一链路：

- 新建 Skill 首轮澄清；
- 已有 Skill 修改 / 优化；
- 导入 Skill 后审计 / 修复。

### 2.3 可接受的设计取舍

用户明确接受以下两类取舍：

- 如果有助于缩短首答耗时，则部分治理 / 审计能力可以改为异步补完；
- 如果有助于满足 SLA，则复杂请求可以采用“两段式回答”，即先交付可执行首答，再续跑深度结果。

## 3. 根因判断

结合现有实现，当前首答偏慢的主要结构性原因不是单点性能问题，而是“重链路串行化”：

- 进入主回答前，可能先做 workflow bootstrap、memo / recovery 装载、sandbox remediation 绑定等同步工作；
- `run_stream` 首 token 前要先做 prompt 构建、历史裁剪、上下文拼装；
- optimize / audit 场景把审计、治理卡片、阶段推进与主回答耦在一条链路里；
- 前后端分别维护部分状态真相，导致 hydration、解析和回填逻辑重复。

如果不先拆掉这条串行重链路，则前端再怎么补反馈，都只是遮挡，不是真正提速。

## 4. 总体架构

### 4.1 双通道执行模型

本设计采用 `Fast Lane + Deep Lane` 双通道架构：

- `Fast Lane`
  - 负责在 `30 秒 / 60 秒` 目标内交付首份可用反馈；
  - 使用轻量路由、轻量上下文、轻量或中等模型；
  - 不阻塞在完整审计、完整治理、完整 remediation 上。

- `Deep Lane`
  - 负责补全复杂审计、治理卡片、staged edits、深层推理、补充证据；
  - 允许异步续跑；
  - 结果必须增量回填当前会话，而不是变成孤立后台任务。

### 4.2 统一状态层

前后端围绕统一的 `run / workflow state` 运转。

同一轮请求必须拥有统一的：

- `run_id`
- `workflow_id`
- `complexity_level`
- `execution_strategy`
- `fast_status`
- `deep_status`

前端不再根据零散事件猜进度，而是基于统一状态展示当前执行阶段。

### 4.3 SLA 驱动的降级机制

如果预计无法在目标窗口内等到完整结果，则系统必须主动降级，而不是继续同步等待：

- 中等复杂请求优先满足 `30 秒首答`；
- 高复杂请求优先满足 `60 秒首答`；
- 未完成部分自动转入 Deep Lane；
- 首答以“两段式回答”交付，而不是继续阻塞整轮请求。

## 5. 统一阶段机

所有 Skill Studio 场景共用同一阶段机，不再由场景分支各自定义完整执行路径：

- `accepted`
- `classified`
- `fast_running`
- `fast_completed`
- `deep_pending`
- `deep_running`
- `deep_completed`
- `blocked`
- `failed`
- `superseded`

场景差异只体现在复杂度、默认策略和 Deep Lane 触发条件上，不体现在是否必须同步跑完整链路。

## 6. 模型分层策略

模型调用拆成三层：

- `L0 路由层`
  - 规则与轻量状态判定；
  - 不依赖重模型；
  - 负责场景识别、复杂度分级、执行策略选择。

- `L1 首答层`
  - 负责首份可用反馈；
  - 使用首答最小上下文；
  - 优先满足交互 SLA。

- `L2 深答层`
  - 负责复杂审计、治理、staged edits、深度 architect 推演；
  - 允许异步续跑；
  - 结果以 patch 形式回填。

## 7. 上下文分层与缓存策略

### 7.1 上下文分层

所有输入上下文拆成三层：

- `Core Context`：当前用户消息、最小历史、skill 核心信息、workflow 摘要；
- `Extended Context`：memo 摘要、recovery 摘要、最近测试摘要、source file 索引；
- `Deep Context`：完整 memo / recovery / 长历史 / 治理卡片 / 文件正文。

`Fast Lane` 只允许使用 `Core Context` 与必要的 `Extended Context` 摘要。  
`Deep Lane` 才允许进入 `Deep Context`。

### 7.2 缓存与摘要

为了避免多轮对话重复付费，系统需要显式建设以下摘要缓存：

- `Conversation Context Digest Cache`
- `Skill Runtime Snapshot Cache`
- `Memo Digest Cache`
- `Recovery Digest Cache`

缓存目标不是保存更多原始数据，而是生成首答层可直接消费的最小摘要。

## 8. 异步化边界

### 8.1 必须同步完成的能力

- 场景识别；
- 复杂度判断；
- 缺失条件判断；
- 主要风险判断；
- 当前建议下一步；
- 首份可执行结论 / 方案 / 修复方向。

### 8.2 可以异步补完的能力

- 深度审计评分；
- 完整治理卡片；
- staged edits 明细；
- remediation 扩展建议；
- architect 深层补充材料；
- 长上下文下的补充风险与证据。

### 8.3 可后台化的附属能力

- 统计型重算；
- memo 非关键整理；
- recovery 归档；
- 低优先级补充卡片与附属说明。

## 9. 回填机制

Deep Lane 的结果不是整块覆盖，而是以 patch 形式回填。标准 patch 类型包括：

- `deep_summary_patch`
- `audit_patch`
- `governance_patch`
- `staged_edit_patch`
- `workflow_patch`
- `evidence_patch`

同一轮请求必须带版本控制：

- `run_version`
- `patch_seq`

如果用户发起了新一轮请求，则旧 run 可被标记为 `superseded`，其后续 patch 不得覆盖当前主状态。

## 10. 前端展示原则

前端第二阶段只能展示与后端真实阶段一一对应的状态，不允许用纯时间驱动的伪进度。

前端必须让用户明确区分：

- 当前是否已经拿到首答；
- Deep Lane 是否仍在运行；
- 当前等待是否还有价值；
- 当前看到的 patch 属于哪一轮 run。

主聊天流承载首答；审计、治理、edits 等深层补丁应落到固定区域，而不是把主消息流冲散。

## 11. 成功标准

如果本设计落地成功，则应满足以下验收口径：

- 中等复杂请求：大部分场景能在 `30 秒` 内交付首份可用反馈；
- 高复杂请求：大部分场景能在 `60 秒` 内交付首份可用反馈；
- 首答达标不依赖假进度提示；
- 审计 / 治理 / remediation 不再成为首答的统一阻塞项；
- 用户能明确知道当前 run 是否已完成首答、是否仍在补完深层结果。

## 12. 不做事项

本轮治理不以“一次性把所有 Agent 基建完全重写”为目标，也不直接引入全量作业平台化重构。

第一阶段重点是：

- 先把 Skill Studio 从同步重链路中解耦出来；
- 先让 SLA 真正可达；
- 再在第二阶段把前端流式协议和中间态展示升级为真实阶段驱动。

## 13. 结论

如果以真实时延治理为目标，则 Skill Studio 后续不能继续沿用“单条重链路串行跑到底”的结构。

可行的总体架构是：

- 以 `Fast Lane + Deep Lane` 作为运行模型；
- 以 `统一阶段机 + 统一 run 状态` 作为协议底座；
- 以 `模型分层 + 上下文分层 + 异步回填` 作为提速手段；
- 以 `真实状态驱动前端中间态` 作为第二阶段交互升级基础。
