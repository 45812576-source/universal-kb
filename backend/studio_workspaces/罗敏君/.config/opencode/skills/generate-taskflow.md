# Generate TaskFlow

## Overview

基于彼得·德鲁克目标管理（MBO）框架，将项目目标分解为多人可执行的结构化任务流。

**核心原则：** 每项任务有且仅有一个负责人，每个交付物有明确可量化的验收标准，结果导向而非过程导向。

## Phase 1: Information Gathering（信息收集）

在生成任务流之前，**必须**向用户提问以下3个问题：

1. **项目目标与背景**：这个项目/活动的核心目标是什么？预期达成什么业务结果？
2. **团队与资源**：参与人员有哪些（姓名/角色）？有哪些外部资源或预算限制？
3. **时间节点**：项目整体截止日期是什么？有哪些关键里程碑或硬性时间节点？

> **提示**：如果用户已提供部分信息，只需追问缺失的部分。

## Phase 2: Strategic Decomposition（战略分解）

收集信息后，按以下5步分解项目：

1. **定义成功标准** — 项目完成时，什么状态代表成功？用可量化指标描述
2. **识别关键阶段** — 将项目拆分为不超过10个逻辑阶段（如：准备期、执行期、收尾期）
3. **分解阶段任务** — 每个阶段拆解为具体可执行的任务（最小粒度可到小时，最大不超过3天）
4. **分配单一负责人** — 每项任务指定唯一负责人（不允许"共同负责"）
5. **识别依赖关系** — 标注哪些任务必须在其他任务完成后才能开始

## Phase 3: Task Definition Standards（任务定义标准）

| 字段 | 定义 | 示例 |
|------|------|------|
| 任务 | 动词开头的具体行动描述 | 设计新员工入职手册 |
| 负责人 | 唯一责任人（姓名或角色） | HR专员-小李 |
| 截止日期 | 具体日期（YYYY-MM-DD） | 2024-03-15 |
| 交付物 | 可见、可验证的产出物 | 入职手册PDF（含10个模块） |
| 验收标准 | 可量化的完成判断依据 | 经3名新员工试读，满意度≥4.5/5 |
| 前置依赖 | 必须先完成的任务编号 | T-01, T-02 |

## Phase 4: Output Format（输出格式）

为每个阶段生成以下格式的任务表：

### 阶段一：[阶段名称]（[开始日期] - [结束日期]）

| 编号 | 任务 | 负责人 | 截止日期 | 交付物 | 验收标准 | 前置依赖 |
|------|------|--------|----------|--------|----------|----------|
| T-01 | [任务描述] | [负责人] | YYYY-MM-DD | [交付物] | [验收标准] | - |
| T-02 | [任务描述] | [负责人] | YYYY-MM-DD | [交付物] | [验收标准] | T-01 |

> 按需重复以上表格，每个阶段一张表。

### 里程碑汇总

| 里程碑 | 目标日期 | 负责人 | 完成标志 |
|--------|----------|--------|----------|
| [里程碑名称] | YYYY-MM-DD | [负责人] | [可量化完成标志] |

## Drucker Principles Checklist（德鲁克原则核查）

交付前逐项确认：

- [ ] 每项任务有且仅有**一个**负责人
- [ ] 所有验收标准均可量化（避免"完成"、"做好"等模糊表述）
- [ ] 任务描述以动词开头，描述具体行动而非状态
- [ ] 交付物是可见、可触摸的产出（文档、数据、系统、活动）
- [ ] 依赖关系已完整标注，无循环依赖
- [ ] 时间节点考虑了缓冲时间（关键路径任务建议留10-20%缓冲）

## Common Mistakes（常见错误）

| 错误 | 示例 | 修正 |
|------|------|------|
| 共同负责 | 负责人：小李、小王 | 负责人：小李（小王协助） |
| 模糊验收标准 | 验收标准：活动顺利举办 | 验收标准：参与人数≥50人，满意度≥4分/5分 |
| 任务粒度过大 | 任务：完成全年招聘计划 | 拆分为：制定招聘需求表、发布JD、筛选简历等子任务 |
| 缺少交付物 | 交付物：（空） | 交付物：招聘需求确认表（含岗位、人数、时间） |
| 任务描述模糊 | 任务：跟进入职事宜 | 任务：完成新员工系统账号开通（含邮箱、OA、代码仓库） |
