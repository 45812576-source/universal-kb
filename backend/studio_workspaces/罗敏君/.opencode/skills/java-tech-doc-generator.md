# Java 技术设计文档生成器

根据飞书需求文档链接，生成标准化的 Java 技术设计文档。

## 触发条件

**用户给出飞书文档链接，并要求生成技术设计文档时，自动调用此 Skill。**

触发示例：
- "帮我生成技术设计文档：https://feishu.cn/docx/xxx"
- "这是需求文档链接，帮我写一下技术设计文档"
- "根据这个飞书文档生成技术设计文档"
- 用户提供飞书链接 + 项目路径 + "生成技术设计文档"

## 执行流程

### 第一步：解析飞书文档链接

使用 `feishu_doc` 或 `feishu_wiki` 工具读取飞书文档内容：
- 如果是 `docx` 文档（/docx/），使用 `feishu_doc` 的 `read` action
- 如果是 `wiki` 文档（/wiki/），使用 `feishu_wiki` 的 `get` action

> **注意**：如果文档无法读取（返回错误），尝试：
> 1. 使用浏览器打开该链接
> 2. 让用户复制文档内容

### 第二步：获取项目路径

从用户输入中提取项目路径。用户可能以以下方式提供：
- "项目在 F:\tianji\commerce-parent\storefront-service"
- "项目路径：/path/to/project"
- 直接提供项目目录

> 如果用户没有提供项目路径，**必须询问用户**项目代码的位置

### 第三步：分析需求内容

从飞书文档中提取：
- **需求背景**：业务背景、功能需求、非功能性需求
- **核心功能点**：需要开发/修改的功能列表
- **数据结构变化**：新增表、修改字段、删除表
- **接口设计**：需要新增/修改的 API 接口
- **业务流程**：核心业务逻辑

### 第四步：了解项目结构

扫描用户提供的项目目录，了解：

1. **模块结构**：
   - `controller/` - REST API 控制器
   - `service/` - 业务服务接口和实现
   - `mapper/` - MyBatis Mapper 接口
   - `entity/` - 数据库实体类
   - `dto/` - 数据传输对象
   - `vo/` - 视图对象
   - `enums/` - 枚举类
   - `config/` - 配置类

2. **技术栈**（从 pom.xml 或 build.gradle 读取）：
   - 框架版本（Spring Boot版本）
   - 数据库（MySQL/PostgreSQL）
   - 缓存（Redis）
   - 消息队列（RabbitMQ/Kafka）
   - 文件存储（OSS/S3）

### 第五步：分析现有代码

根据需求分析结果：

1. **如果是新增表**：
   - 查找相似的现有表结构作为参考
   - 确定表名、字段、索引设计
   - 查找相关的实体类作为模板

2. **如果是修改现有功能**：
   - 查找相关的 Service、Controller、Mapper
   - 分析现有的业务逻辑
   - 确定需要修改的代码位置

3. **如果是新增接口**：
   - 查找相似的现有接口作为参考
   - 确定接口路径、请求/响应格式

### 第六步：生成技术设计文档

按照标准格式生成 Markdown 文档，保存到 `D:\openclaw\article-workflow\{需求名称}-技术设计文档.md`

1. 生成完整的技术设计文档
2. 文档保存到 `D:\openclaw\article-workflow\{需求名称}-技术设计文档.md`
3. 确保文档内容完整、专业、可落地执行

## 注意事项

- 如果飞书文档链接无法访问，**必须使用浏览器打开**，或让用户复制内容
- **必须询问用户项目路径**，不能硬编码
- 如果项目代码结构不清晰，先向用户确认
- 对于复杂的业务流程，建议在文档中加入时序图（mermaid）
- 生成文档前，确保已充分理解需求和现有代码
